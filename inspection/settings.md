# Project Settings and Runtime Inspection

## Scope

Inspected project-level runtime/configuration files:

- `crypto_track/settings.py`
- `crypto_track/urls.py`
- `crypto_track/asgi.py`
- `crypto_track/wsgi.py`
- `manage.py`
- `requirements.txt`
- `README.md`
- `.gitignore`
- `inspection/README.md` for report consistency

I also read the app URL files and the direct settings consumers in `coins/services.py`
and `coins/management/commands/runapscheduler.py` to understand how project settings
are used at runtime. This was inspection only; no application code was modified.

## Executive Summary

The project uses a single Django 5.2.5 settings module with mostly startproject
defaults plus PostgreSQL URL parsing, database-backed cache, crispy Bootstrap 5,
django-apscheduler, and required CoinGecko configuration.

The current configuration is workable for local development when a populated
virtualenv, PostgreSQL role/database, cache table, and `CRYPTO_COINGECKO_KEY`
exist. It is brittle for fresh setup and unsafe for production as-is:

- `crypto_track/settings.py` requires `CRYPTO_COINGECKO_KEY` at import time.
- Production mode requires `CSRF_TRUSTED_ORIGINS` at import time.
- `ALLOWED_HOSTS = ["*"]` disables meaningful host validation.
- The default `SECRET_KEY` is a committed Django insecure development key.
- `STATIC_ROOT` is missing, so production `collectstatic` is not configured.
- HTTPS/security cookie settings are absent.
- `manage.py` loads `.env`, but `asgi.py` and `wsgi.py` do not, so process manager
  environments must provide all variables explicitly.
- The active shell Python cannot import Django; the checked-in `venv/` can.

## Settings Module Structure

`crypto_track/settings.py` is a single environment-switching module rather than
separate development/test/production modules.

- It was generated by `django-admin startproject` using Django 5.2.5
  (`crypto_track/settings.py:1`).
- `BASE_DIR` is the repository root, computed with `Path(__file__).resolve().parent.parent`
  (`crypto_track/settings.py:13`, `crypto_track/settings.py:19`).
- Runtime configuration is read directly from `os.environ`, with
  `dj_database_url` handling database URL parsing (`crypto_track/settings.py:13`,
  `crypto_track/settings.py:16`).
- Environment mode is inferred from `PYTHON_ENV != "production"` rather than
  Django's more common explicit `DJANGO_DEBUG`, `DJANGO_ENV`, or separate
  settings module pattern (`crypto_track/settings.py:31`).

There is no typed settings layer, no `.env.example`, no production settings file,
and no validation helper that can report all missing configuration at once.

## Installed Apps and Middleware

`INSTALLED_APPS` combines Django contrib apps, project apps, form rendering, and
scheduler persistence (`crypto_track/settings.py:40`):

- Django contrib: admin, auth, contenttypes, sessions, messages, staticfiles.
- Project apps: `common`, `coins`, `accounts`, `portfolio`.
- Third-party apps: `crispy_forms`, `crispy_bootstrap5`, `django_apscheduler`.

The installed app order is conventional enough for this project. `common` is
installed before app templates that consume its template tags/partials, and
`django_apscheduler` enables scheduler job/execution models.

`MIDDLEWARE` is the standard Django stack (`crypto_track/settings.py:60`):

- `SecurityMiddleware`
- session middleware
- common middleware
- CSRF middleware
- auth middleware
- messages middleware
- clickjacking middleware

Notable omissions for production:

- No static-file serving middleware such as WhiteNoise.
- No custom middleware for request IDs, logging, error reporting, or security
  headers beyond Django defaults.
- No middleware that would compensate for missing HTTPS proxy settings.

## Templates and Static Files

Templates use Django's template backend with a root `templates/` directory and
app-template discovery enabled (`crypto_track/settings.py:72`).

- `DIRS = [BASE_DIR / "templates"]` enables shared project templates such as the
  global shell.
- `APP_DIRS = True` enables app-local templates.
- Context processors include `request`, `auth`, and `messages`
  (`crypto_track/settings.py:78`). Existing common pagination templates rely on
  `request`.
- Crispy forms are configured for Bootstrap 5 with
  `CRISPY_ALLOWED_TEMPLATE_PACKS = "bootstrap5"` and
  `CRISPY_TEMPLATE_PACK = "bootstrap5"` (`crypto_track/settings.py:56`).

Static configuration is development-oriented:

- `STATIC_URL = "static/"` (`crypto_track/settings.py:146`).
- `STATICFILES_DIRS = [BASE_DIR / "static"]` (`crypto_track/settings.py:147`).
- `STATIC_ROOT` is not defined, so production `collectstatic` has no configured
  destination.
- There is no asset pipeline, manifest storage, CDN configuration, WhiteNoise
  configuration, or cache-busting static storage.

This matches the current lightweight Bootstrap/static CSS setup, but it should be
made explicit before deployment.

## Database and Cache

The default database uses `dj_database_url.config()` (`crypto_track/settings.py:93`).

- `CRYPTO_DATABASE_URI` is optional.
- The fallback is `postgres://crypto_track@/crypto_track`
  (`crypto_track/settings.py:95`).
- README setup expects a local PostgreSQL role and database named `crypto_track`
  (`README.md:38`).
- Tests also require privileges on the `postgres` database according to the README
  (`README.md:44`).

There is no SQLite development fallback, so a fresh local run requires PostgreSQL
even for basic app usage and tests. There is also no explicit database connection
pooling, SSL option, test database override, or production-only connection tuning.

The default cache is Django's database cache (`crypto_track/settings.py:101`):

- Backend: `django.core.cache.backends.db.DatabaseCache`.
- Table name: `cache`.
- README correctly requires `python manage.py createcachetable`
  (`README.md:52`).
- CoinGecko supported-coin list timeout is two hours in seconds:
  `CACHE_SUPPORTED_COINS_TIMEOUT = 3600 * 2` (`crypto_track/settings.py:108`).
- Market page cache timeout is 60 seconds (`crypto_track/settings.py:109`).

Runtime implications:

- Missing the `cache` table can break CoinGecko cache access at request time.
- The cache shares the primary database, so high market-data traffic adds database
  reads/writes rather than using an external cache.
- The scheduler consumes `CACHE_SUPPORTED_COINS_TIMEOUT` through
  `coins.services.SUPPORTED_COINS_TIMEOUT` (`coins/services.py:10`), but the
  scheduler then treats that seconds value as APScheduler minutes
  (`coins/management/commands/runapscheduler.py:50`). That is an app-level bug
  caused by unclear setting units.

## Auth, Sessions, and Security Settings

Auth is stock Django auth:

- `django.contrib.auth` is installed (`crypto_track/settings.py:42`).
- Auth middleware and context processor are enabled (`crypto_track/settings.py:65`,
  `crypto_track/settings.py:80`).
- Default password validators are enabled (`crypto_track/settings.py:115`).
- No custom `AUTH_USER_MODEL` is configured, so the project uses `auth.User`.
- No `LOGIN_URL`, `LOGIN_REDIRECT_URL`, or `LOGOUT_REDIRECT_URL` project setting is
  defined. App URL/view code hard-codes login/logout destinations instead
  (`accounts/urls.py:6`, `accounts/urls.py:8`, `accounts/views.py:12`).

Security-sensitive settings:

- `SECRET_KEY` reads `DJANGO_SECRET`, but falls back to a committed
  `django-insecure-...` key (`crypto_track/settings.py:26`).
- `DEBUG` is `False` only when `PYTHON_ENV == "production"`
  (`crypto_track/settings.py:32`).
- `ALLOWED_HOSTS = ["*"]` in all modes (`crypto_track/settings.py:34`).
- `CSRF_TRUSTED_ORIGINS` is empty in debug and required from the environment in
  production (`crypto_track/settings.py:35`).
- Clickjacking middleware is enabled, so Django's default `X_FRAME_OPTIONS =
  "DENY"` applies unless changed elsewhere (`crypto_track/settings.py:67`).

Missing production hardening:

- `SECURE_SSL_REDIRECT`
- `SECURE_PROXY_SSL_HEADER`
- `SESSION_COOKIE_SECURE`
- `CSRF_COOKIE_SECURE`
- `SECURE_HSTS_SECONDS`
- `SECURE_HSTS_INCLUDE_SUBDOMAINS`
- `SECURE_HSTS_PRELOAD`
- secure `ALLOWED_HOSTS`
- structured secret/config validation

`venv/bin/python manage.py check --deploy` confirms Django reports security
warnings for HSTS, SSL redirect, insecure secret key, secure session cookie, and
secure CSRF cookie when run with production-like variables.

## Environment Variables and Deployment Assumptions

Environment variables currently used:

- `DJANGO_SECRET`: optional secret key override (`crypto_track/settings.py:27`).
- `PYTHON_ENV`: only the exact value `production` disables debug
  (`crypto_track/settings.py:32`).
- `CSRF_TRUSTED_ORIGINS`: required only when `PYTHON_ENV=production`
  (`crypto_track/settings.py:35`).
- `CRYPTO_DATABASE_URI`: optional database URL (`crypto_track/settings.py:95`).
- `CRYPTO_COINGECKO_KEY`: required at settings import
  (`crypto_track/settings.py:157`).
- `DJANGO_SETTINGS_MODULE`: set by startup entrypoints if absent
  (`manage.py:13`, `crypto_track/asgi.py:14`, `crypto_track/wsgi.py:14`).

README documents `CRYPTO_COINGECKO_KEY` and `CRYPTO_DATABASE_URI`, and says
`.env` files are supported (`README.md:64`). `.gitignore` excludes `.env`
(`.gitignore:4`).

Important mismatch:

- `manage.py` calls `load_dotenv()` before importing Django (`manage.py:3`).
- `asgi.py` and `wsgi.py` do not call `load_dotenv()`
  (`crypto_track/asgi.py:10`, `crypto_track/wsgi.py:10`).

That means local management commands can read `.env`, but ASGI/WSGI deployments
must supply environment variables through the process manager, platform, or shell.
This is usually acceptable for production, but the README should make the split
clear.

## CoinGecko Configuration

CoinGecko configuration is project-level:

- `COINGECKO_ENDPOINT = "https://api.coingecko.com/api/v3/"`
  (`crypto_track/settings.py:156`).
- `COINGECKO_KEY = os.environ["CRYPTO_COINGECKO_KEY"]`
  (`crypto_track/settings.py:157`).

`coins/services.py` reads these settings into module-level constants at import
time (`coins/services.py:8`). A thread-local `requests.Session` adds the key as
the `x-cg-demo-api-key` header (`coins/services.py:16`).

Runtime implications:

- Missing `CRYPTO_COINGECKO_KEY` prevents any Django settings import, even for
  unrelated commands.
- Rotating the key in a running process will not update already-imported service
  module constants or existing thread-local sessions.
- Endpoint and key configuration are not namespaced in a service object, which
  makes tests and future API-plan changes harder.
- The settings expose cache timeouts but not HTTP timeout, retry, rate-limit, or
  response-validation policy.
- The variable name says `KEY`, while the header uses CoinGecko's demo key header;
  if the app moves to a paid/pro endpoint, the endpoint/header pairing should be
  revisited.

## Startup Entrypoints

`manage.py`:

- Loads `.env` immediately (`manage.py:3`).
- Sets `DJANGO_SETTINGS_MODULE` to `crypto_track.settings` if absent
  (`manage.py:13`).
- Delegates to Django's `execute_from_command_line()` (`manage.py:15`).
- Raises Django's standard import-help message when Django is unavailable
  (`manage.py:17`).

`crypto_track/asgi.py` and `crypto_track/wsgi.py`:

- Set `DJANGO_SETTINGS_MODULE` to `crypto_track.settings` if absent
  (`crypto_track/asgi.py:14`, `crypto_track/wsgi.py:14`).
- Expose module-level `application` objects from `get_asgi_application()` and
  `get_wsgi_application()` (`crypto_track/asgi.py:16`, `crypto_track/wsgi.py:16`).
- Do not load `.env`.

There is no custom startup behavior for scheduled jobs, websocket routing, health
checks, metrics, or static serving. The APScheduler process is started manually
through a management command documented in the README (`README.md:71`).

## URL Routing Overview

The root URL config is small (`crypto_track/urls.py:21`):

- `/admin/` routes to Django admin (`crypto_track/urls.py:22`).
- `/` includes `coins.urls` (`crypto_track/urls.py:23`).
- `/accounts/` includes `accounts.urls` (`crypto_track/urls.py:24`).
- `/portfolio/` includes `portfolio.urls` (`crypto_track/urls.py:25`).

Mounted app routes:

- `coins.urls` owns the market index, search, watchlist, and watchlist toggle
  routes (`coins/urls.py:11`).
- `accounts.urls` owns login, register, and logout (`accounts/urls.py:5`).
- `portfolio.urls` owns overview, add/edit/delete transaction, and all
  transaction routes (`portfolio/urls.py:6`).

The project does not set `handler404`, `handler500`, health check routes, API
routes, static/media serving routes for development, or namespaced account URLs.

## Dependency and Runtime Implications

`requirements.txt` pins runtime and developer dependencies together:

- Django 5.2.5 and asgiref 3.9.1
  (`requirements.txt:9`, `requirements.txt:2`).
- PostgreSQL driver: `psycopg2-binary` (`requirements.txt:18`).
- Database URL parsing: `dj-database-url` (`requirements.txt:7`).
- Forms: `django-crispy-forms` and `crispy-bootstrap5`
  (`requirements.txt:11`, `requirements.txt:6`).
- Scheduling: APScheduler and django-apscheduler
  (`requirements.txt:1`, `requirements.txt:10`).
- HTTP: `requests` plus urllib3/certifi/idna/charset-normalizer
  (`requirements.txt:22`, `requirements.txt:26`, `requirements.txt:3`,
  `requirements.txt:14`, `requirements.txt:4`).
- `.env`: `python-dotenv` (`requirements.txt:19`).
- Linting/formatting/import sorting: `ruff` (`requirements.txt:23`,
  `pyproject.toml`, `.pre-commit-config.yaml`).
- Git hook orchestration: `pre-commit` and its support packages
  (`requirements.txt:17`, `.pre-commit-config.yaml`).

Implications:

- There is no separate development/test/production dependency split.
- `ruff`, `pre-commit`, and hook support packages are installed in production if
  `requirements.txt` is used directly.
- There is no explicit Python version file, but the local interpreters checked
  during inspection are Python 3.12.13.
- There is no ASGI/WSGI server dependency such as gunicorn, uvicorn, or daphne.
- There is no test runner dependency beyond Django's built-in test framework.
- There is no Redis/memcached dependency, matching the database cache choice.

## Local Development and Test Friction

Fresh setup requires several external pieces before the app can run:

1. Create/activate a virtualenv and install requirements (`README.md:30`).
2. Create a PostgreSQL role/database (`README.md:38`).
3. Set `CRYPTO_COINGECKO_KEY` (`README.md:64`).
4. Run migrations (`README.md:46`).
5. Create the database cache table (`README.md:52`).
6. Run `runapscheduler --run-now` to seed local `Coin` rows (`README.md:71`).

Friction points:

- Active shell `python manage.py check` fails because Django is not installed in
  the active interpreter.
- The checked-in `venv/bin/python` can run Django checks, so contributors need to
  know which interpreter to use.
- Any settings import needs a CoinGecko key, even for commands that do not call
  CoinGecko.
- PostgreSQL is mandatory by default; there is no quick SQLite fallback for
  isolated UI or unit-test work.
- Database cache requires an extra management command.
- Tests likely require database privileges described in the README and can still
  hit live CoinGecko flows unless mocked at app level.
- `.env` support only applies to `manage.py`, not ASGI/WSGI process imports.

Runtime checks attempted with `CRYPTO_COINGECKO_KEY=dummy`:

- `python manage.py check`: failed with `ModuleNotFoundError: No module named 'django'`
  in the active interpreter.
- `venv/bin/python manage.py check`: passed with no issues.
- `PYTHON_ENV=production CSRF_TRUSTED_ORIGINS=https://example.com venv/bin/python manage.py check --deploy`:
  completed with five Django security warnings.
- `PYTHON_ENV=production venv/bin/python manage.py check`: failed with
  `KeyError: 'CSRF_TRUSTED_ORIGINS'`.

## Production Risks

High-priority production risks:

1. Wildcard `ALLOWED_HOSTS` accepts any host header (`crypto_track/settings.py:34`).
2. The fallback secret key is committed and explicitly insecure
   (`crypto_track/settings.py:26`).
3. `CRYPTO_COINGECKO_KEY` and production `CSRF_TRUSTED_ORIGINS` can crash settings
   import with raw `KeyError` (`crypto_track/settings.py:35`,
   `crypto_track/settings.py:157`).
4. HTTPS redirect, secure cookies, HSTS, and proxy SSL settings are absent.
5. `STATIC_ROOT` and production static serving strategy are absent.
6. Database cache can become a bottleneck and is easy to misconfigure by missing
   `createcachetable`.
7. There is no process model documented for web server plus scheduler worker.
8. There is no deployment dependency for a WSGI/ASGI server.
9. `DEBUG` depends on `PYTHON_ENV` exactly equaling `production`; typoed values
   silently enable debug.
10. Production settings are mixed with development defaults in one module, making
    it easy to ship development behavior.

## Refactor and Polish Opportunities

Concrete settings/runtime improvements for the refactor pass:

1. Split settings into `base`, `development`, `test`, and `production`, or add a
   small typed config layer that validates required environment variables with
   clear error messages.
2. Replace `PYTHON_ENV != "production"` with an explicit `DJANGO_DEBUG` parser
   that fails closed in production environments.
3. Require `DJANGO_SECRET` outside development and remove the committed fallback
   from any production path.
4. Make `ALLOWED_HOSTS` environment-driven and never wildcard in production.
5. Parse `CSRF_TRUSTED_ORIGINS` safely with trimming, empty-value handling, and
   scheme validation.
6. Add production HTTPS/security settings, including secure cookies, HSTS,
   proxy SSL handling, and explicit documentation for reverse-proxy assumptions.
7. Add `STATIC_ROOT`, choose a static serving strategy, and consider manifest
   storage or WhiteNoise for simple deployments.
8. Decide whether local/test environments should use SQLite, PostgreSQL, or a
   documented containerized PostgreSQL setup; encode that choice.
9. Consider replacing database cache with Redis/memcached for production, or make
   the database-cache tradeoff explicit.
10. Rename/cache timeout settings with units, such as
    `CACHE_SUPPORTED_COINS_TIMEOUT_SECONDS`, to prevent scheduler unit bugs.
11. Move CoinGecko settings into a small client/config object with API key,
    endpoint, timeout, retry/rate-limit policy, and test overrides.
12. Avoid mandatory CoinGecko key lookup at settings import for commands that do
    not use CoinGecko.
13. Align `.env` behavior across management commands and ASGI/WSGI, or document
    that only `manage.py` loads local `.env` files.
14. Add `.env.example` with required and optional variables.
15. Split runtime and development requirements, and add a production server
    dependency or deployment docs.
16. Add a health check route and production logging/error reporting configuration.
17. Centralize auth redirect settings instead of hard-coding login/logout targets
    in app URL/view code.
18. Document the expected process topology: web process, scheduler process,
    database, cache, and static files.
